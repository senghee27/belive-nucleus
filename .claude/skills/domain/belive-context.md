## Cell Operating Model (v3) — Core Architecture

### The Three-Role Cell

```
IOE = Brain    → Case owner, priority setter, tenant comms, cell coordinator
OOE = Body     → Physical execution, inspection, turnaround, room readiness
Technician = Hands → Repair execution, fix quality, SLA compliance
```

Each cluster runs as a cell. IOE leads daily execution but is NOT the HR boss of OOE or Tech.

**Functional reporting:**
- IOE → IOE Lead → Fatihah (OM)
- OOE → Adam (OOE Lead) → Fatihah (OM)
- Tech → Fariha (Maintenance Manager)

**Operational daily coordination:** IOE sets priority → OOE + Tech execute

### Old vs New Model

```
OLD (broken): Tenant → IOE informs OOE → OOE informs Maint → Wait
             Result: handoff friction, blame shifting, slow resolution

NEW (cell):   Tenant → IOE owns case → classifies & prioritises
              → OOE/Tech execute → IOE closes loop with tenant
             Result: single accountable owner, internal pressure replaces management pressure
```

## Role Definitions

### IOE — Indoor Operation Executive (Cell Leader / Case Owner)
- Portfolio: 300–500 rooms per IOE
- Reports to: IOE Lead → Fatihah (OM)

**Five cognitive modes:**
1. Tenant Outcome Owner — case is mine until tenant confirms closure
2. Cell Coordinator — know what's open, urgent, who's doing what
3. Priority Controller — classify by: 1) safety/habitability, 2) move-in/out/OR impact, 3) owner escalation risk, 4) service quality, 5) low urgency QOL
4. Closed-Loop Communicator — issue only solved when tenant confirms closure
5. Early Risk Sensor — detect non-renewal risk, frustration, hidden patterns

**KPIs:**
- Case Resolution SLA: 25%
- Communication Quality: 15%
- Lifecycle Management: 20%
- Aging Control: 15%
- Cell Coordination: 15%
- Recurrence Rate: 10%

**Escalation paths:**
- OOE issue → Adam
- Tech blocked → Fariha
- Cross-functional → Fatihah
- Safety → immediate OM

**Identity:** NOT an admin coordinator or complaint forwarder. IS a tenant-facing case owner and cell execution lead.

### OOE — Outdoor Operation Executive (Physical Execution Engine)
- Portfolio: 350–450 rooms per OOE
- Reports to: Adam (OOE Lead) → Fatihah (OM)
- Takes daily priority from: IOE

**Turnaround pipeline:**
```
Confirm ≤12h → Inspect ≤24h → Classify → Clean ≤24h → Repair 24-48h → Verify → Sales Ready

**Five cognitive modes:**
1. Execution Mode — is this room physically ready right now?
2. Inspection Mode — identify and classify hidden issues
3. Coordination Mode — align Tech + Cleaning + IOE for correct sequence
4. Risk Detection Mode — spot delays before IOE escalates
5. Recovery Mode — unstick pipeline, get rooms back to sales fast

**KPIs:**
- Turnaround Time: 35% (target ≤48h)
- Move-In Success: 20% (target ≥99%)
- Inspection SLA: 15% (target ≤12h)

**Identity:** NOT a runner or inspection staff. IS a turnaround owner and physical execution engine.

```

## Red Flags to Watch For

- Cases bounced around without a single owner
- Tenant chasing for updates (IOE should be proactive)
- "In progress" but no real movement or next step
- Fake closure without tenant confirmation
- IOE blaming OOE/Tech instead of owning
- Rooms stuck in turnaround >48h
- Cherry-picking easy jobs, ghost completions
- Rising backlog with no daily task list posted
- Tech posting only minor/easy tasks while critical tickets pile up

## Scan Framework — 7 Things to Check Per Cluster

1. Is there an owner for every issue? (IOE accountability)
2. Is there movement or is it stuck? (SLAs: IOE closure, OOE ≤48h, Tech ≤48h)
3. Is the tenant being updated? (IOE closed-loop comms)
4. Is anyone blaming instead of owning? (cell model breakdown)
5. Are rooms stuck in turnaround pipeline? (OOE execution)
6. Are there repeat repairs or rising backlogs? (Tech quality)
7. Who should be tagged? (IOE for case, Adam for OOE, Fariha for maintenance, Fatihah for cross-functional)

## Core Business Model — 7 Steps

**Step 1 — Source/onboard owner**
Direct acquisition, developer channels, agency/referral, renovation-to-rental programs, bundled offers

**Step 2 — Assess co-living feasibility**
Room count potential, layout efficiency, tenant demand, yield per room, capex requirement, occupancy potential

**Step 3 — Renovate / fit out**
Layout optimization, room partitioning, built-in furniture, durability planning, Spacify smart system installation. Renovation logic is NOT cosmetic-first — designed around durability, maintenance practicality, tenant behavior, ROI, and smart system compatibility.

**Step 4 — Launch to market**
Listing, marketing, lead generation

**Step 5 — Fill rooms**
Room-by-room monetization vs single whole-unit rental → significantly higher gross yield

**Step 6 — Operate the property**
Collection, tenant support, cleaning coordination, maintenance, renewal handling, utilities, smart lock/meter management, owner reporting, issue resolution, turnover handling

**Step 7 — Repeat and scale**
More units, more rooms, standardized systems
